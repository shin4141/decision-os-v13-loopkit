## 3. Delivery Scope Radar

Delivery Scope Radar is a client-delivery diagnostic derived from Entry Window Radar.

Its purpose is:

> To decide how far to build for a client project, given price, deadline, review value, and repeat-business potential.

Example:

* a 50,000 yen job on Coconala
* one-week deadline
* desire for repeat business and good review
* over-service would turn the work unprofitable
* AI can build it, but the correct thickness is unclear

Delivery Scope Radar would output:

* **Minimum Delivery Line**
  The point below which the delivery is too weak.

* **Review Boost Line**
  The point that improves satisfaction, review, and repeat-business potential.

* **Diminishing Return Start**
  The point after which more effort stops converting well into value.

* **Over-Service Zone**
  Too much. Risk of loss, late delivery, and depletion.

* **Next Price Anchor**
  What to preserve from this delivery as evidence for raising the next price.

## 4. Genre-Specific Diminishing-Return Start

Delivery Scope Radar depends on genre-specific points where effort begins to diminish.

More polish does not always mean more value.

Examples:

* In landing-page work, first view, CTA, mobile display, and contact path tend to matter.
* Excess animation or overly fine UI can fall after the diminishing-return start depending on project price.
* In AI automation, one automation used every day can matter.
* Generic settings screens or excessive edge-case handling may become over-service in a 50,000 yen project.

Core judgment:

> Delivering the used range quickly and carefully is more valuable than building unused high functionality.

This is also a practical form of Resource Justice.

## 5. Resource Justice / V14 Elements

For V13, Entry Window Radar is a future-line discovery device.

Adding V14 Resource Justice changes the question.

V13 asks:

> Which future line may be opening?

V14 asks:

> Should this person allocate finite resources to that future line?

An entry window being open and the operator being able to carry that window are different.

`FAST ENTRY` can become fast depletion if handled badly.

## 6. V14 Variables

Future Pro / Advanced layers may handle the following Resource Justice variables.

### Resource Budget Fit

Can the current Carrier pay the time, money, body, attention, token, and validation count required?

### Opportunity Cost Drag

Does betting on this future line pull resources away from the line that is currently growing best?

### Proof Cost

What evidence cost is required to move from an interesting idea to a believable artifact?

Inspect:

* whether README alone is enough
* whether samples are needed
* whether CLI is needed
* whether real-user diagnostic examples are needed
* whether GitHub stars are needed
* whether third-party review is needed
* whether before / after evidence is needed
* whether a paid path is needed

### Runway-to-Signal Fit

Can attention, body, tokens, and improvement drive be maintained until the market signal arrives?

### Value Ceiling

Is there a ceiling where more effort no longer increases trust, adoption, stars, consultation, or revenue?

### Branch Debt

Opening a repo or future line creates issue handling, README updates, posts, reaction checks, guilt when abandoned, and re-entry cost.

### Re-entry Capacity

If the attempt fails, can the operator return to the main line?

If not, the result is not `FAST ENTRY`; it is `CAP` or `AVOID`.

## 7. Free / Pro Boundary

Resource Justice elements are strong, but putting all of them into the free OSS version makes the product too heavy.

Boundary:

### Free

As-of fixed-point diagnosis.

* current position
* Entry Window Chart
* `FAST ENTRY / NICHE WEDGE / WAIT / SHORT CYCLE / AVOID`
* strengths
* missing pieces
* one biggest bottleneck
* one next move
* confidence
* missing evidence
* light Resource Note

The free version should move the user forward.

Example:

> As of now, this does not reach FAST ENTRY.
> But NICHE WEDGE is possible.
> The biggest bottleneck is distribution path.
> First, do not add features. Get one reaction where the expected user already is.

### Pro / Advanced

Track lines and trajectories.

* Score History
* Entry Window Drift
* Operator Edge trajectory
* Aspire / Goal-Length changes
* Proof Cost
* Resource Budget Fit
* Runway
* Value Ceiling
* Branch Debt
* Re-entry Capacity
* whether effort is affecting the score
* whether there is enough slack to rest
* whether to hurry

The free version shows the current point.
The Pro version shows lines and trajectories.

## 8. Mock-Exam Frame and Paid Path

Entry Window Radar is closer to a mock exam than to the real university entrance exam.

The mock exam is not the real exam.
People still take it because they want to know whether their effort is moving toward the target.

Entry Window Radar is the same.

The free version answers:

> Where is the current fixed point?

The Pro version answers:

> Over time, is effort riding the right line?

People do not buy prediction.
They buy relief that their effort is not accumulating in the wrong direction.

## 9. Evidence Addition and Re-Evaluation

The first score is not a fixed verdict.
It is an as-of snapshot.

The user can add material and request re-evaluation.

Flow:

```text
Initial As-of Score
-> Missing Evidence
-> Add Evidence
-> Rescore
-> Entry Window Shift
```

This matters.

In a previous case, a new Claude chat initially treated the user as a beginner.
As papers and artifacts were shown, the discussion became sharper.

The same applies to Entry Window Radar.

* First run: low confidence because material is sparse.
* Added material: past articles, repos, achievements, and publishing history.
* Re-evaluation: Operator Edge and confidence change.
* Pro version: track the shift.

In the future, this trajectory can become a paid value point.

## 10. Entry Window Drift

In the Pro version, the movement can matter more than the absolute score.

Example:

Last week:

* Operator Edge: 58
* Entry Window: 45 days
* distribution evidence weak

This week:

* Operator Edge: 66
* Entry Window: 59 days
* artifact evidence increased and distribution hypothesis clarified

Interpretation:

> Several days of slack were created. Rest is possible. If pushing further, add one distribution path.

Opposite example:

Last week:

* Entry Window: 60 days

This week:

* Entry Window: 42 days

Reasons:

* competitors increased
* similar OSS appeared
* major-player absorption risk rose

Interpretation:

> Get public reaction before further polishing.

This produces both FOMO and relief.

## 11. Aspire / Goal-Length Connection

From V10, the tool should not judge only the user's stated surface goal.

For example, the user may say:

> side-income

But underneath, the real aspiration may be:

> to work long-term in children, education, or support fields

In that case, short-term revenue may be `HOLD`, while long-term Aspire foundation may be `GO / CAP`.

The Pro version may inspect:

* Stated Goal
* Underlying Aspire
* Survival Bound
* Goal-Length
* whether current effort rides the Aspire line
* whether Goal-Length shortened or lengthened

Do not overload the free version with this.
Handle it in Pro / Advanced.

## 12. Minimal Resource Note in Free Version

Do not put all V14 elements into the free version.
A light Resource Note can still be included.

Example:

> There is an entry window, but this is a small-bet stage, not an all-in stage.
> Start with cheap evidence: README, one sample, or one post.

Another example:

> This idea is attractive, but Proof Cost is high.
> Before implementation, it is safer to get market reaction.

The free Resource Note should not stop the user.
It should encourage a smaller bet.

## 13. Delivery Scope Radar Is Not the Main Line

Delivery Scope Radar is promising, but it should not be mixed into the initial Entry Window Radar MVP.

Reasons:

* It leans toward client and delivery projects.
* Its use case differs from repo / README entry-window diagnosis.
* It blurs the initial README focus.
* It sits slightly away from the GitHub stars / forks path.

Preserve it as a future derivative candidate.

Positioning:

> Entry Window Radar: where to enter.
> Delivery Scope Radar: how far to go on a project already accepted.

Shared core:

> Find the range where effort converts into value.
