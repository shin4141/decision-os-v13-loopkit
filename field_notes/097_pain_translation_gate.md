# Pain Translation Gate

## Observation

V13 ideas can become clean before they become usable.

A concept may be internally coherent, well named, and compatible with previous field notes, but still fail to answer a user's immediate question:

```text
What pain does this reduce for me now?
```

When the pain translation is weak, the next action often drifts toward explanation, promotion, or system expansion.

That drift can look productive while avoiding the harder test: whether the idea maps to a concrete felt problem.

## Core claim

Before promoting, publishing, productizing, or broadening a V13 concept, translate it into user pain.

```text
No pain translation, no promotion.
```

This is not a marketing rule.

It is a next-loop gate.

## What pain translation means

Pain translation means converting an internal V13 distinction into a plain user-facing before/after.

Examples:

```text
Before:
I keep re-explaining the same repo state to every AI session.

After:
The next AI can restart from the handoff without guessing.
```

```text
Before:
My agent says the work is done, but I do not know if the next task should continue.

After:
V12 checks restartability, and V13 decides whether the next loop is GO, HOLD, CAP, or BLOCK.
```

```text
Before:
My AGENTS.md is full of good instructions, but the AI still behaves inconsistently.

After:
Check whether the file is short, routed, and behaviorally legible, not only whether it is shorter.
```

## Gate question

Use this before broadening a concept:

```text
Can this be stated as a concrete pain, failed moment, or repeated cost that a user would recognize without learning V13 first?
```

If yes:

- continue only with the smallest matching surface
- prefer a copy-paste prompt, example, service note, or bounded field note
- preserve the boundary and stop condition

If no:

- keep the concept parked
- record it as a seed if it may matter later
- do not rewrite README
- do not promote to AGENTS.md / CLAUDE.md
- do not create automation, hooks, MCP, pluginization, or an execution engine

## Difference from explanation

Explanation says:

```text
Here is what the concept means.
```

Pain translation says:

```text
Here is the repeated cost this prevents.
```

The second is the stronger test.

If a concept cannot survive this translation, it may still be intellectually useful, but it is not ready to become an entrypoint, offer, public post, or canonical instruction.

## Why this matters

V13 has many strong internal distinctions:

- V12 vs V13
- Execution Health vs Capability Reserve
- restartability vs completion
- router vs warehouse
- complexity vs length
- field-note seed vs current task
- public exposure vs internal proof

These distinctions become externally useful only when they reduce an experienced pain:

- re-explaining context
- losing the next safe action
- trusting clean completion too much
- bloated always-loaded instructions
- unreadable old repo state
- public posting before evidence
- concept growth without a concrete trigger

Pain translation keeps V13 from becoming self-referential.

## Relationship to Capability Reserve

When Capability Reserve is `YELLOW`, pain translation should be especially strict.

A long context can still generate clean concepts.

It should not continue into heavy development unless the next pain, surface, and stop condition are small and clear.

## One-sentence summary

The Pain Translation Gate prevents clean internal V13 concepts from becoming promotion, product, or system expansion until they map to a concrete user-recognizable pain.
