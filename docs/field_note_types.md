# V13 Field Note Types

V13 LoopKit uses field notes to record proof-of-use.

Not all field notes are the same.  
This note distinguishes three field note types used in the prototype.

## 1. Self-Application

Self-Application means applying V13 LoopKit to V13 LoopKit itself.

It asks:

> Should this project’s own next loop GO, HOLD, CAP, or BLOCK?

Use Self-Application when the project is improving its own operating surface.

Example:

- deciding whether to add proof-of-use before public promotion
- deciding whether the repo should add features or stay documentation-only
- deciding whether the next loop should be a field note, prompt, schema, or README change

Self-Application is useful because it shows the theory controlling its own growth.

Risk:

Self-Application can become too meta if it never touches a concrete task.

Boundary:

A Self-Application note is valid only when it produces a concrete next action or cap.

## 2. Real-Task Proof

Real-Task Proof means applying V13 LoopKit to a concrete completed task.

It asks:

> After this real task was completed, what should the next loop do?

Use Real-Task Proof when an AI-assisted task has actually changed files, produced artifacts, published something, or completed a work unit.

Example:

- Codex updated AGENTS.md and README.md
- an AI coding task created a schema and examples
- a writing task produced a publishable draft
- a research task created a note or citation structure

Real-Task Proof is the most important evidence type for V13 LoopKit.

It shows whether the gate is useful, unclear, too heavy, or wrong in real operation.

Risk:

A real-task note can become too heavy if every small edit receives a full field note.

Boundary:

Use Real-Task Proof only for tasks that produce reusable residue or change the next loop.

## 3. Public-Exposure Control

Public-Exposure Control means applying V13 LoopKit to decisions about posting, announcing, promoting, or increasing visibility.

It asks:

> How much public exposure is allowed in the next loop?

Use Public-Exposure Control when the next action may consume attention, credibility, trust, or public positioning.

Example:

- deciding whether to announce a new paper
- deciding whether to post a GitHub repo
- deciding whether to write follow-up explanations
- deciding whether to respond to weak public reaction

Public-Exposure Control is not marketing planning.

It is Carrier protection.

Risk:

Public-exposure loops can become reaction-chasing loops.

Boundary:

A Public-Exposure Control note should usually output CAP or HOLD unless there is strong proof-of-use and clear audience fit.

## Quick Classification

| Question | Field Note Type |
|---|---|
| Are we using V13 on V13 itself? | Self-Application |
| Did a concrete task complete and now needs next-loop review? | Real-Task Proof |
| Is the next action public posting, promotion, or visibility? | Public-Exposure Control |

## Prototype Rule

If uncertain, prefer Real-Task Proof.

Self-Application is useful for governing the project.  
Public-Exposure Control is useful for protecting the Carrier.  
Real-Task Proof is the main evidence that V13 LoopKit works.

## Minimal Decision Rule

Use:

- Self-Application when the project itself is the object.
- Real-Task Proof when a completed task is the object.
- Public-Exposure Control when public attention is the object.

In one line:

> Self-Application governs the project, Real-Task Proof tests the tool, and Public-Exposure Control caps the visibility loop.
