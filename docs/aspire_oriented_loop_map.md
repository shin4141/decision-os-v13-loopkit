# Aspire-Oriented Loop Map

## Purpose

A defensive Loop Map prevents drift.

An Aspire-Oriented Loop Map asks a second question:

> Is this loop moving the operator closer to the declared Aspire without damaging the Carrier?

V13 does not only decide whether a loop should continue.
It decides whether the loop is moving the operator closer to the declared Aspire without damaging the Carrier.

## Why This Matters

A loop can be safe but still useless.

A loop can be productive locally but irrelevant to the upper objective.

A loop can generate activity without moving toward adoption, stars, revenue, or operationalization.

Therefore, V13 must distinguish:

- loops that protect the Carrier
- loops that reduce friction
- loops that move toward stars or adoption
- loops that observe real data
- loops that convert observed gaps into 0.01 improvements
- loops that must stop until Carrier recovers

## Phase Model

### Phase 1: Defense

First, protect the system.

Examples:

- V12 completion integrity
- V13 GO / HOLD / CAP / BLOCK
- public exposure caps
- feature growth pause
- v1.0 hold
- irreversible action protection
- ownership-sensitive human decision gates

### Phase 2: Approach

Once defense is stable, move toward Aspire.

Examples:

- reduce first-use friction
- make AGENTS.md copy-paste useful
- make the README easier to understand
- create a simple demo
- clarify the first star-acquisition path
- make the tool usable without requiring humans to manually write Loop Records

### Phase 3: Observation

Observe whether the loop actually moves the project.

Examples:

- GitHub stars
- views
- clones
- forks
- issues
- comments
- external reuse
- README confusion
- user drop-off
- repeated questions

### Phase 4: 0.01 Correction

Only correct exposed gaps.

A 0.01 loop should be created when a gap is exposed by:

- user confusion
- Codex confusion
- missing output
- repeated unclear reports
- README friction
- public misunderstanding
- insufficient conversion
- missing decision surface
- observed metric weakness

Do not create speculative work only because something might be useful later.

### Phase 5: Carrier Recovery

If the operator or project is being depleted, pause.

Examples:

- public reaction-chasing
- too many repo edits without proof
- feature growth without adoption
- repeated posts without signal
- pushing toward v1.0 before enough evidence
- reducing Aspire by over-optimization

Carrier recovery is not failure.

It is a required condition for compounding.

## Aspire-Oriented Loop Types

| Loop | Purpose | Aspire Contribution | Default Gate |
|---|---|---:|---|
| Defense Loop | prevent damage and false completion | High | GO |
| Proof Loop | create real-task evidence | High | GO / CAP |
| Friction Reduction Loop | reduce first-use effort | High | GO / CAP |
| Star Acquisition Loop | increase public adoption signal | High | CAP |
| Observation Loop | collect evidence from real use | High | GO |
| Conversion Loop | turn attention into adoption | Medium / High | CAP |
| Revenue Loop | move toward monetization | High | HOLD until evidence |
| Carrier Recovery Loop | protect long-term continuation | High | GO when needed |
| Feature Growth Loop | add capability | Conditional | HOLD unless exposed gap |
| V1 Paper Loop | formalize theory | High later | HOLD until evidence |

## Example: V13 Lite Footer

Exposed gap:

> V13 Loop Records are too manual. V12 felt easier because AGENTS.md could be copied and immediately produced a signal.

0.01 improvement:

> Add a V13 Lite Footer to AGENTS.md so Codex automatically reports the next loop gate after the V12 completion state.

Aspire contribution:

> High. It reduces first-use friction and makes the tool easier to star, reuse, and understand.

Gate:

> GO / CAP. Add the footer, but do not add automation, CLI, server, or external integrations.

## 0.01 Rule

A 0.01 loop is valid when:

1. The gap has been exposed by use, user reaction, real task, or observed data.
2. The improvement moves toward declared Aspire.
3. The change does not damage Carrier.
4. The change does not violate the current Loop Map.
5. The change improves the next loop's starting condition.

One-line rule:

> Do not improve everything. Improve the smallest exposed gap that moves the project closer to Aspire.
