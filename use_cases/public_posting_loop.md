# Use Case: Public Posting Loop

Use this when a completed work state creates possible public posts, release notes, essays, updates, or social fragments.

The danger is exposure without governance: publishing may create signal, but it can also drain attention, credibility, trust, or recovery capacity.

V13 decides whether public posting should continue, wait, run under a cap, or stop.

## Typical Completed State

- A project, decision, research note, or product update is complete.
- There is public-facing residue worth sharing.
- The next loop would publish, test messaging, or increase distribution.
- Engagement, fatigue, or reputation effects may be uncertain.

## V13 Question

Given the completed work and public residue, should the next public posting loop be GO, HOLD, CAP, or BLOCK?

## Copy-Paste Input

Paste the completed work summary and draft posts into [`../prompts/v13_loop_review.md`](../prompts/v13_loop_review.md).

Add this context:

```text
Use case: public posting loop.

Pay special attention to:
- whether the post creates useful signal or only motion
- whether public exposure is bounded
- whether the loop preserves attention and credibility
- whether replies and follow-up obligations are manageable
- whether the operator can re-enter work after publishing
```

## Gate Examples

- GO: publish when the message is clear, low-risk, useful, and does not create unmanaged follow-up burden.
- HOLD: wait when audience, claim strength, or review burden is unclear.
- CAP: publish only under a concrete limit on time, number of posts, channel, or automation.
- BLOCK: stop when publishing damages trust, credibility, attention, or re-entry capacity.

## Example CAP

```text
Run 3 posts max over 48 hours. No automation. No paid boost. Recheck engagement, fatigue, and reply burden before continuing.
```

## Example Next Loop Command

```text
Publish only the first bounded post, collect signal for 24 hours, then complete a new V13 Loop Record before posting again.
```
